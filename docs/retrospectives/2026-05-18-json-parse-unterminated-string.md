# JSON Parse: Unterminated String Retrospective

**Status: unresolved — investigation paused**

## Context

When using the app (exact trigger not yet confirmed), the API returned:

```json
{"detail":"Unterminated string starting at: line 152 column 16 (char 12401)"}
```

This is FastAPI's standard error format for a `json.JSONDecodeError` raised while parsing the **request body**. The JSON parser hit the end of input (or an invalid character) while still inside a string value, at character offset 12401 in the payload.

## What Was Traced

The error surfaces from FastAPI's built-in request body validation. The `{"detail": "..."}` wrapper is generated by FastAPI, not application code. "Line 152, column 16" refers to line 152 of the JSON document being parsed — not Python source line 152.

A 12401-character JSON document at the point of failure points to one of the heavier endpoints:

- **`POST /api/guides`** — carries `images: string[]` (base64 PNG per PDF page)
- **`POST /api/session`** — carries the full `Guide` object (all steps, all image URLs) plus an optional base64 JPEG photo

In local dev mode (`BLOB_LOCAL_FALLBACK=1` or no valid token), `blob.py` returns a full `data:image/png;base64,...` URL instead of a Vercel CDN URL. Those data URLs can be hundreds of KB each. When the guide is sent back in session requests, every step with an image embeds its full PNG data URL in the JSON body.

## Hypotheses Explored

**Body truncation (leading hypothesis)**
If a proxy, Vercel, or Mangum layer enforces a request body size limit and truncates the payload, Python's json parser would see a string that opens with `"` but never closes — producing exactly this error. The 12401-character cutoff is suspicious because it is specific rather than a clean power-of-two limit.

**Raw control characters in string values**
`JSON.stringify` in JavaScript escapes all U+0000–U+001F control characters. However, pre-ES2019 environments do not escape U+2028 (LINE SEPARATOR) or U+2029 (PARAGRAPH SEPARATOR). Python's `json.loads` rejects these as raw characters inside a string. If step content extracted by Claude contains these Unicode line terminators, and the user's browser is pre-ES2019, the serialized JSON would be technically invalid and Python would reject it.

**Lone Unicode surrogates**
JavaScript strings (UCS-2) can contain lone surrogate code points (`\uD800`–`\uDFFF`). `JSON.stringify` serializes them as `\uD8xx` sequences, which are not valid JSON. Python's `json.loads` would raise a decode error. The Web Speech API `transcript` field is a plausible source for malformed Unicode input.

**BYOK changes introducing a body issue**
The recent `require_api_key` dependency reads from `request.headers`, not the body. It was ruled out — it cannot affect body parsing.

## What Was Ruled Out

- `apiFetch` — correctly sets the `X-Api-Key` header and forwards the body unchanged
- `JSON.stringify` call sites in `VoiceLoop`, `ProgressCheck`, and `GuideCreator` — all correct
- The API key header itself interfering with body parsing — headers and body are independent

## Root Cause

Not yet confirmed. The leading candidate is body truncation caused by large `image_url` data URLs being round-tripped in session requests during local development.

## Prevention (if root cause is data URL round-trip)

- Strip `image_url` fields from the `Guide` object before sending it in session requests — the session API does not use them, only the frontend does for display
- Or resolve images to CDN URLs even in local dev by using a mock blob store

## Prevention (if root cause is Unicode in step content)

- In `api/utils.py`, sanitize Claude's output in `extract_json` to remove or escape U+2028, U+2029, and lone surrogates before returning step content
- Consider using `ensure_ascii=True` in any Python JSON serialization of Claude output

## Learning

A FastAPI "Unterminated string" error on the request body means the Python JSON parser ran out of document before closing a string. This always means one of two things: the payload was physically truncated (size limit), or it contains a raw character that is illegal inside a JSON string. `JSON.stringify` is not a guarantee of valid JSON — pre-ES2019 browsers and strings containing lone surrogates or Unicode line terminators can produce output that fails Python's stricter parser.

When diagnosing, check the payload size first using browser DevTools before chasing encoding bugs.
